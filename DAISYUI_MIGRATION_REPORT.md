# DaisyUIへの完全移行に関する調査レポート

本プロジェクト（Mono - Markdown to Monolithic Document）のすべてのCSSをDaisyUIへ完全に移行した場合の影響について調査した結果を報告します。

## 1. 作業範囲と作業内容

プロジェクト内のすべてのCSSをDaisyUIに移行するためには、以下の作業が必要になります。

- **グローバルスタイルの置換**: 現在の `src/templates/core/base.css` および `themes.css` を削除し、DaisyUIおよびTailwind CSSに置き換える。
- **Web Componentsの改修**:
  - `src/components/` 以下のすべてのコンポーネント（現在20種類以上）の `style.css` を破棄し、`template.html` 内のHTMLタグにDaisyUI/Tailwindのユーティリティクラス（`btn`, `card`, `text-primary` など）を直接記述するよう書き換える。
- **テーマ切り替えの統合**: 現在の `mono-theme` コンポーネントのロジックを、DaisyUIの標準テーマシステム（`data-theme` 属性の切り替え）に適合させる。
- **ビルドパイプラインの構築**:
  - 現在のPython (`uv`) 完結のツールチェーンに、Node.js / Tailwind CLIのビルドプロセスを追加する。
  - ユーザーがMarkdownをHTMLに変換する際、動的に生成されたHTMLに対してTailwindのPurge（不要クラスの削除）を実行する仕組みを導入する。

## 2. メリットとデメリット

### メリット
- **デザインの統一性と品質向上**: カスタムCSSを書かずに、洗練された一貫性のあるUI（ボタン、カード、モーダルなど）を簡単に実現できる。
- **豊富なテーマ**: DaisyUIが提供する多数のビルドインテーマを即座に利用できる。
- **開発速度の向上**: ユーティリティファーストな設計により、新規コンポーネント作成時のCSS実装コストが大幅に下がる。

### デメリット
- **単一ファイル出力（ファイルサイズ）の大幅な肥大化**:
  - 本ツールは「すべてが埋め込まれた自己完結型HTML」を生成します。DaisyUIの全スタイルをそのまま埋め込むとファイルサイズが数MB増加します。
- **Shadow DOMとの致命的な相性の悪さ**:
  - Web Components（Shadow DOM）は外部のグローバルCSSを遮断します。そのため、Shadow DOM内でDaisyUIのクラスを機能させるには、**すべてのコンポーネントの `<style>` タグ内にDaisyUIの巨大なCSSを重複して注入**する必要があります。これによりHTMLサイズが爆発的に増加します。
- **ビルドプロセスの複雑化**: Pythonツールチェーンのみで動くという手軽さが失われ、Node.js環境のセットアップやJITコンパイルの実行が必要になります。

## 3. DaisyUI移行後のMarkdownの記述方法の変化について

- **HTMLタグとクラスの直接記述が増加する**:
  - これまで単純なマークダウンやカスタム記法（例: `@[row]` のようなレイアウトコンポーネント）で記述していたものが、DaisyUIの恩恵を最大限に受けるために、ユーザー自身が `<div class="card bg-base-100 shadow-xl">...</div>` のように直接HTMLとクラスを記述する場面が増える可能性があります。
- **カスタムコンポーネントの記法は維持可能**:
  - `mono-poll` や `mono-ab-test` などのWeb Componentは内部でDaisyUIクラスを使用するよう改修されるため、Markdown上での呼び出し方自体は変わりません。

## 4. その他、懸念事項

- **オフライン要件の喪失リスク**:
  - ファイルサイズの肥大化を防ぐためにDaisyUIをCDNから読み込む設計に変更した場合、ネットワーク環境がない場所（オフライン）でドキュメントを開いた際にCSSが当たらず、完全に表示が崩れてしまいます。これは「自己完結型」という本ツールのコアバリューに反します。
- **パフォーマンスの低下**:
  - 各Web ComponentのShadow DOMごとに同一の巨大なスタイルシートを注入すると、ブラウザのパース時間およびメモリ使用量が増大し、特に低スペックな端末やモバイルでの表示が重くなる懸念があります（Constructable Stylesheets を用いるなどの高度なハックが必要になりますが、単一ファイルHTMLへの書き出しでは適用が困難です）。
- **Markdownパース時のTailwind JITコンパイルの限界**:
  - ユーザーがMarkdown内で自由にTailwindクラスを記述した場合、事前にビルドされたCSSにはそのクラスが含まれていない可能性があります。変換ツール実行時（Python側）にNode.jsを呼び出して動的にCSSを生成・パージ処理を行うのは、実行速度と依存環境の両面で非現実的です。

---
**結論**:
現在の「単一HTMLにすべてを埋め込む」「Vanilla JSのWeb Components (Shadow DOM) を多用する」というアーキテクチャにおいて、DaisyUI (Tailwind CSS) への完全移行は、技術的制約（特にShadow DOMによるスタイル隔離とファイルサイズの問題）により、非常に高いハードルとデメリットを伴います。
