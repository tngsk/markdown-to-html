# DESIGN.md 導入調査レポート

## 1. DESIGN.md の概要
`DESIGN.md` は、Google Labs が公開している「コーディングエージェント（AI）に視覚的アイデンティティ（デザインシステム）を伝えるためのフォーマット仕様」です。
マシンリーダブルなデザイントークン（YAMLフロントマター）と、人間が読めるデザインの意図やルール（Markdownの本文）を一つのファイルに結合しています。

- **YAML部分**: カラーコード（`#1A1C1E`等）、タイポグラフィ、余白（Spacing）、角丸（Rounded）、コンポーネントごとのスタイルなどを定義。
- **Markdown部分**: 「なぜその色を使うのか」「どのようにコンポーネントをレイアウトすべきか」といったコンテキストや「Do's and Don'ts」を記述。
- **ツール**: `npx @google/design.md` CLIを通じて、構造のLint（WCAGコントラスト比のチェック、未定義トークンの検出など）、バージョン間の差分抽出、他フォーマット（TailwindやDTCGなど）へのエクスポート機能が提供されています。現在は `alpha` バージョンです。

## 2. 本プロジェクト（Mono）への活用可能性
本プロジェクト「Mono」はMarkdownから単一のHTMLを生成し、Vanilla JSベースのWeb Componentsを用いてリッチな表現を提供しています。スタイリングにおいては `src/templates/core/themes.css` 内にDaisyUIインスパイアのCSS変数を定義し、テーマシステムを構築しています。

本プロジェクトはAIエージェント（LLM）を用いた開発を積極的に取り入れており、UIコンポーネントの作成やデザインの統一において、`DESIGN.md` の概念は非常に親和性が高く、**十分に活用可能**であると考えられます。

## 3. 具体的な活用方法
Monoにおいて、以下のような形での導入・活用が考えられます。

1. **AIエージェント向けのデザインガイド（リポジトリへの配置）**
   リポジトリのルートに `DESIGN.md` を配置し、AIエージェントが新規Web Component（例: `mono-button` など）を実装・修正する際の「プロンプトの一部（コンテキスト）」として機能させます。これにより、エージェントが勝手なハードコードの色（hexやrgb）を使用するのを防ぎ、`themes.css` のCSS変数を正しく利用するように誘導できます。

2. **CSSの自動生成（テーマシステムのソース・オブ・トゥルース化）**
   `DESIGN.md` のYAMLフロントマターを正と見なし、ここからビルド時に `src/templates/core/themes.css` を自動生成するスクリプトを導入します。これにより、デザイン定義と実装（CSS）の乖離を防ぎます。

3. **エンドユーザー向けカスタムテーマ入力機能**
   MonoのCLIツールの引数として、Markdownファイルだけでなくユーザー独自の `DESIGN.md` を渡せるように拡張します（例: `uv run main.py document.md --design DESIGN.md`）。変換時にYAMLを解析し、単一HTMLに埋め込むインラインCSSとして動的にテーマ（CSS変数）を生成して適用することが可能です。

## 4. 導入のメリット・デメリット

### メリット
- **AIによる実装精度の向上**: AIエージェントがデザイン意図（なぜそのスタイルなのか）と正確な値（トークン）を同時に把握できるため、MonoのWeb Component開発においてデザインのブレがなくなります。
- **ドキュメントと定義の一元化**: 従来分離しがちだった「デザインシステムの説明書（ドキュメント）」と「実際の定義値（変数）」が1ファイルにまとまるため、メンテナンス性が向上します。
- **アクセシビリティの確保**: 公式のLinterを通すことで、コントラスト比（WCAG AA）などのアクセシビリティ検証を自動化できます。

### デメリット
- **仕様の不確実性**: `DESIGN.md` 自体がまだ `alpha` 版であり、今後スキーマやルールが破壊的に変更されるリスクがあります。
- **既存のテーマ構造との統合コスト**: 現在の `themes.css`（ライト/ダーク/コーポレート等のDaisyUIライクな構成）を `DESIGN.md` の構造（Primary/Secondary/Tertiary/Neutral等）にマッピング・移行する作業が発生します。
- **二重管理のリスク**: CSSの自動生成機構を導入しない場合、`DESIGN.md` と `themes.css` の両方を手動で更新する必要があり、二重管理に陥る可能性があります。

## 結論
現状でも、AIエージェントへのコンテキスト共有用ファイル（ガイドライン）として `DESIGN.md` をリポジトリに配置するだけであれば、導入コストなしで大きなメリット（AIの出力安定化）が得られます。
一方で、システムに組み込む（CSS生成やユーザー入力対応など）場合は、`DESIGN.md` フォーマットの成熟度を見極めつつ、段階的な統合を検討するのが望ましいです。
