# デスクトップアプリケーション化の設計書

本プロジェクトのマルチプラットフォーム向けデスクトップアプリケーション化に向けたアプローチ、技術選定、および機能実現のアーキテクチャ設計を以下にまとめます。

## 1. アプローチと技術選定

現在のシステム（PythonによるMarkdownから自己完結型HTMLへの変換ロジック、およびFastAPIベースのサーバー機能）を最大限活用しつつ、マルチプラットフォーム（Windows, macOS, Linux）対応のアプリケーションを構築するために、以下の技術を推奨します。

### 第一候補: **PyWebView**
- **特徴**: Pythonのみで完結し、OS標準のWebエンジン（WindowsのWebView2、macOSのWebKitなど）を使用してネイティブウィンドウ内にHTML/JSベースのUIを描画する軽量ライブラリ。
- **メリット**:
  - 現在のPythonコードベースに直接統合でき、追加の言語（RustやNode.js）を必要としない。
  - フロントエンドとPythonバックエンド間で、JS API (`window.pywebview.api`) を使ったシームレスな通信が可能。
  - 既存のFastAPIサーバーを別スレッドで立ち上げることも、Webサーバーなしで直接ファイル処理関数を呼び出すことも可能。
- **デメリット**: OSのWebエンジンに依存するため、環境によって微妙なレンダリングの差異が出る可能性がある。

### 第二候補: **Tauri (Python Sidecar パターン)**
- **特徴**: Rustベースの軽量・高速なフレームワーク。フロントエンドはWeb技術（HTML/CSS/JS）、バックエンドはRustだが、Pythonスクリプトを「Sidecar」としてバンドルして連携可能。
- **メリット**: 非常に軽量でセキュア、OSネイティブに近い操作感。
- **デメリット**: ビルドプロセスが複雑（RustとPython/PyInstallerの両方の管理が必要）。

**結論**: 既存のPythonエコシステムを維持し、迅速な開発・移行を優先するため、**PyWebView**を主軸としたアーキテクチャを推奨します。

## 2. アーキテクチャ設計

### 全体構成 (PyWebView採用時)
- **バックエンド (Python)**:
  - 既存の変換ロジック (`main.py`, `processors`, `embedders` 等)
  - `pywebview` によるウィンドウ管理とAPIブリッジ
  - （必要に応じて）バックグラウンドでのFastAPIサーバー実行
- **フロントエンド (HTML/CSS/JS)**:
  - アプリケーションのUIシェル。
  - **DaisyUI**（Tailwind CSSプラグイン）を使用し、モダンでフレンドリーなUIを構築。

### 機能要件の実現方法

#### ① Markdown読み込み（開く、ドラッグ＆ドロップ）
- **開く（ダイアログ）**:
  - UI上の「ファイルを開く」ボタンをクリックすると、JSからPythonのAPIを呼び出し、`webview.windows[0].create_file_dialog()` を実行。ネイティブのファイル選択ダイアログを表示し、選択されたファイルパスを取得します。
- **ドラッグ＆ドロップ**:
  - HTML5のDrag & Drop APIを使用し、UI上にドロップゾーンを作成。
  - ドロップされたファイルのパスを取得し、Python側へ送信して変換処理を開始します。

#### ② プレビュー機能
- 変換処理が十分に高速であることを活かし、プレビュー機能を実装します。
- **アプローチ**:
  - ユーザーがファイルを選択またはドロップした直後、バックグラウンドでPythonがHTML変換を実行。
  - 生成されたHTML文字列をJSに返し、プレビュー用の `<iframe>` の `srcdoc` 属性、または Shadow DOM 内に注入して表示します。
  - 複雑化を避けるため、リアルタイムのタイピングプレビューではなく、「ファイル読み込み時のプレビュー」程度のシンプルな構成から始めるのが適当です。

#### ③ 設定管理画面の構築（DaisyUI + Markdown）
- **UIコンセプト**: DaisyUIを利用した親しみやすく直感的な画面。
- **Markdownによる設定画面の実装**:
  - 本システムの「MarkdownからHTMLへの変換機能」を活用します。
  - 設定画面のレイアウトや説明文を `settings.md` として記述し、独自のMarkdown拡張（またはカスタムWeb Component）を使用して、設定用フォーム（例: `@[setting: theme](radio)`）を埋め込みます。
  - アプリケーション起動時にこのMarkdownを変換して設定画面のUIとしてレンダリングします。
  - フォームの入力状態は、JSからPython側のAPIを経由して `config.toml` などの設定ファイルに即座に反映（書き込み）されます。

## 3. 実装のロードマップ（提案）

1. **PoC（概念実証）の作成**:
   - `pywebview` を導入し、シンプルなDaisyUIベースのHTMLファイルを表示するウィンドウを立ち上げる。
2. **APIブリッジの構築**:
   - フロントエンドからPythonの変換処理を呼び出せるようにする。
3. **ファイル操作の統合**:
   - ファイルオープンダイアログとドラッグ＆ドロップの実装。
4. **プレビューと設定画面**:
   - 生成されたHTMLのプレビュー表示エリアの作成。
   - `settings.md` と `config.toml` を連携させる設定管理画面の構築。
5. **パッケージング**:
   - `PyInstaller` などを使用し、各OS向けの単一実行可能ファイル（`.exe`, `.app` など）としてビルドする仕組みを構築。
